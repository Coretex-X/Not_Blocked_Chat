### Настройка Redis (строки 13-19)

```python
redis_pool = ConnectionPool.from_url(
    'redis://localhost:6379/0',
    max_connections=100,
    decode_responses=True,
    socket_keepalive=True
)
```
**Построчное объяснение:**

**Строка 13-14:** `redis_pool = ConnectionPool.from_url('redis://localhost:6379/0',`
- `ConnectionPool.from_url()` — создаёт пул соединений из URL
- `'redis://localhost:6379/0'` — URL подключения к Redis:
  - `redis://` — протокол подключения
  - `localhost` — хост (Redis запущен на той же машине)
  - `6379` — стандартный порт Redis
  - `/0` — номер базы данных Redis (Redis поддерживает до 16 баз данных, нумерация с 0)

**Строка 15:** `max_connections=100,`
- Максимальное количество одновременных соединений в пуле
- Если приложению нужно больше 100 соединений, оно будет ждать освобождения одного из существующих
- Значение 100 достаточно для большинства приложений среднего размера

**Строка 16:** `decode_responses=True,`
- Автоматически декодирует ответы Redis из байтов в строки (UTF-8)
- Без этого все данные возвращались бы как `b'data'` (байты) вместо `'data'` (строки)
- Упрощает работу с текстовыми данными

**Строка 17:** `socket_keepalive=True`
- Включает механизм keep-alive на уровне TCP-сокета
- Периодически отправляет пустые пакеты для проверки, что соединение всё ещё активно
- Помогает обнаружить разорванные соединения и автоматически переподключиться

```python
redis_client = Redis(connection_pool=redis_pool)
```
**Построчное объяснение:**
- `Redis()` — создаёт клиент Redis
- `connection_pool=redis_pool` — указывает использовать ранее созданный пул соединений
- Все операции с `redis_client` будут переиспользовать соединения из пула
- Этот клиент будет использоваться во всех consumer'ах для работы с сессиями

### Метод validate_token_and_get_session

```python
async def validate_token_and_get_session(self, token_session, session_type):
```

**Назначение:** Проверяет валидность токена сессии и возвращает данные сессии из Redis.

**Асинхронный метод:**
- Определён с `async def`, что означает асинхронное выполнение
- Может использовать `await` для ожидания других асинхронных операций
- Не блокирует другие операции во время ожидания ответа от Redis

**Параметры:**
- `self` — ссылка на экземпляр класса
- `token_session` — токен сессии, который нужно проверить (обычно UUID)
- `session_type` — тип сессии: `"new_chat"` (новый чат) или `"existing_chat"` (существующий)

**Возвращает:**
- `dict` с данными сессии (`room`, `user_id`, `guest_id`, `token`) — если токен валиден
- `None` — если токен невалиден, истёк или не найден

**Логика работы:**
1. Ищет сессию в Redis по ключу `session:{session_type}:{token_session}`
2. Проверяет соответствие токена
3. Удаляет сессию из Redis (одноразовое использование)
4. Возвращает данные сессии

# Класс DataConsumer

## Описание класса

```python
class DataConsumer(AsyncWebsocketConsumer):
```

**Назначение:** Consumer для аутентификации и авторизации пользователей перед подключением к чату.

**Наследование:**
- Наследуется от `AsyncWebsocketConsumer` (Django Channels)
- НЕ наследуется от `BaseChatConsumer` — это отдельный, независимый consumer

**Основные функции:**
- Проверяет права доступа пользователя к чат-комнате
- Создаёт временную сессию в Redis с токеном
- Выдаёт одноразовый токен для подключения к реальному чату
- Работает как "шлюз" перед основным чатом

**Архитектурный паттерн:**
1. Клиент подключается к `DataConsumer` с учётными данными
2. `DataConsumer` проверяет права доступа
3. Если доступ разрешён, создаёт временную сессию в Redis (живёт 5 минут)
4. Отправляет клиенту подтверждение
5. Закрывает соединение
6. Клиент использует токен для подключения к `ChatConsumer` или `NewChatConsumer`

**Зачем нужен такой паттерн:**
- Разделение логики аутентификации и коммуникации
- Безопасность: токен одноразовый и живёт 5 минут
- Масштабируемость: аутентификация быстрая, не держит долгие соединения

## Диаграмма работы DataConsumer

```
1. Клиент подключается к /ws/auth/
        ↓
2. DataConsumer.connect() → принимает соединение
        ↓
3. Клиент отправляет JSON с учётными данными
        ↓
4. DataConsumer.receive() → парсит данные
        ↓
5. Проверяет обязательные поля
        ↓
6. Вызывает check_access() для проверки прав
        ↓
7a. Если права есть → создаёт сессию в Redis
        ↓
    Отправляет {"status": "success"}
        ↓
    Закрывает соединение (finally)
        ↓
    Клиент подключается к /ws/chat/{token}/ или /ws/new_chat/{token}/

7b. Если прав нет → закрывает соединение с кодом 4001
        ↓
    Клиент получает ошибку
```

## Сценарии использования

### Сценарий 1: Создание нового чата

**Шаги:**
1. Пользователь A хочет начать чат с пользователем B
2. Клиент генерирует уникальный ID комнаты (например, UUID)
3. Клиент подключается к `DataConsumer`
4. Клиент отправляет:
   ```json
   {
     "room": "new-room-uuid-123",
     "user_id": "userA",
     "guest_id": "userB",
     "status_chat": "new_chat",
     "token": "session-token-456"
   }
   ```
5. `DataConsumer` проверяет, что комната `"new-room-uuid-123"` не существует
6. Создаёт сессию в Redis: `session:new_chat:session-token-456`
7. Отправляет `{"status": "success"}`
8. Закрывает соединение
9. Клиент подключается к `/ws/new_chat/session-token-456/`
10. `NewChatConsumer` проверяет токен, извлекает данные из Redis, создаёт чат

### Сценарий 2: Подключение к существующему чату

**Шаги:**

**Зачем эта проверка:**
- Если токен был невалиден, метод `connect()` завершился на строке 308 (`return`)

- Сценарий 1: Токен валиден → `connect()` выполнился полностью → `room_group_name` создан → условие `True`
- Сценарий 2: Токен невалиден → `connect()` вышел на строке 308 → `room_group_name` НЕ создан → условие `False`

```python
await self.channel_layer.group_discard(self.room_group_name, self.channel_name)
```

**Строка 322:**
- Удаляем consumer из группы чата

- Поддерживает актуальность списка активных участников группы

- Помогает отслеживать активность и диагностировать проблемы

## Диаграмма создания нового чата

```
1. Пользователь A хочет создать чат с пользователем B
        ↓
2. Клиент подключается к DataConsumer
        ↓
3. DataConsumer проверяет, что комнаты не существует
        ↓
4. DataConsumer создаёт сессию "new_chat" в Redis (TTL 5 мин)
        ↓
5. DataConsumer отправляет {"status": "success"}
        ↓
6. DataConsumer закрывается
        ↓
7. Клиент подключается к NewChatConsumer с токеном
        ↓
8. NewChatConsumer валидирует токен (извлекает из Redis)
        ↓
9. NewChatConsumer создаёт ПЕРВУЮ запись в БД:
   UserData(user_id="A", guest_id="B", room="room123")
        ↓
10. NewChatConsumer создаёт ВТОРУЮ запись в БД:
    UserData(user_id="B", guest_id="A", room="room123")
        ↓
11. NewChatConsumer добавляет себя в группу "chat_room123"
        ↓
12. NewChatConsumer принимает WebSocket-соединение
        ↓
13. Чат создан! Теперь можно обмениваться сообщениями
        ↓
14. При следующем подключении к этому чату используется ChatConsumer
```

**Основные характеристики:**
- **Роль:** Создание новых чатов

- **Токен:** Валидирует токен типа `"new_chat"`
- **База данных:** ✅ Создаёт ДВЕ записи (по одной для каждого участника)

**Методы:**
1. `add_chat()` — создаёт запись в БД для одного участника
2. `connect()` — валидирует токен, создаёт записи, добавляет в группу, принимает соединение

```
DataConsumer (проверка, что комнаты нет) → 
создаёт сессию "new_chat" → 
NewChatConsumer (создаёт записи в БД) → 
BaseChatConsumer (обработка сообщений) →
При следующем подключении → ChatConsumer
```

```python
class NotificationConsumer(AsyncWebsocketConsumer):
```
```python
    """

        }))
```

**Параметры:**

**Возвращает:**
- Ничего (неявно `None`)

**Применение:**
- Уведомление участников чата о том, что пользователь подключился
- Уведомление об отключении пользователя
- Отображение статуса "онлайн" в интерфейсе

**Где вызывается:**
- Может вызываться из view при логине/логауте пользователя
- Может вызываться из сигналов Django
- Может вызываться из других consumer'ов

**Важно:**

## Интеграция с существующими consumer'ами

Чтобы функция работала корректно, нужно добавить обработчик в `BaseChatConsumer`:

```python
class BaseChatConsumer(AsyncWebsocketConsumer):
    # ... существующие методы ...
    
    async def user_status(self, event):
        """Обработчик события изменения статуса пользователя"""
        user_id = event.get("user_id")
        status = event.get("status")
        
        # Отправить уведомление клиенту
        await self.send(text_data=json.dumps({
            "type": "user_status",
            "user_id": user_id,
            "status": status
        }))
```

После добавления этого метода:
- `ChatConsumer` и `NewChatConsumer` автоматически получат обработчик (наследование)
